테스트 케이스 구성요소
1. TC ID
 - TC의 고유한 ID로 무결성 보장
 - TC 삭제 시, ID도 삭제(프로젝트마다 처리 방식이 다르며, 삭제 정보를 기록하기 위해 삭제하지 않는 경우도 있음)
 - 테스트 대상 힌트(예시) order_001 → 주문 관련 테스트)
2. Depth(분류)
 - 명확한 분류 기준(예시) 대분류, 중분류, 소분류)
 - 메뉴 트리 이용한 기준
 - 때에 따라 항목명 수정 가능
3. Pre-Condition(사전조건)
 - 테스트 절차 수행 위해 필요한 환경
 - TC 수행 시, 필요한 데이터(예시) 특정 상품 번호, 쿠폰 번호)
 - 예시) 회원 로그인 , (상품이 존재하는)장바구니
4. Test step(스텝)
 - TC 수행하기 위한 절차
 - 예시) 임의의 상품 선택, 주문하기 버튼 선택
5. Expected result(기대결과)
 - 명세에 표기되어 있는 해당 TC의 출력값
 - 구체적인 예상결과 작성
 - 결함의 추적성과 TC유지보수
6. Actual result (수행결과)
 - 기대결과와 비교해 결정
 - 결과값 입력(예시) Pass, Fail, Block, N/A, N/T)
7. Defect ID(BTS ID)
 - 결함관리 툴에서 생성된 결함 ID(결함 등록 후, ID 기재)
 - 결함, TC, 요구사항 간의 추적성 확보
8. Description(or 비고)
 - 결함의 상세한 내용 혹은 특이사항 기입

-------------------------------------------------------

결과값 구성요소
1. NoRun : 수행하지 않음

2. Pass : 정상 동작
3. Fail : 비정상 동작
4. Block : 전 케이스가 fail이 나면서 테스트 할 수 없는 경우

5. N/A : 비유효
 - 환경 문제, 고객사가 제공하지 않는 서비스, 테스트 케이스에 삭제하지 않음(추후 진행)
 - 예시) 테스트 대상의 환경적인 요소로 인해 테스트 불가한 상황
6. N/T : 테스트 제외
 - 기능을 빼기로 하여 제외, 테스트케이스에서 삭제해도 무방
 - 예시) 개발/사업/환경구성 등의 사유로 테스트 불가한 상황
